Feature Branch Workflow

A simple and effective workflow where all feature development takes place in dedicated branches, keeping the `main` branch stable.

Visual Workflow Diagram

main Branch

The main branch contains the official project history and should always be stable and deployable.

  • This is the single source of truth for the project.
  • All new work starts by creating a branch from main.
  • Code is only added to main by merging completed feature branches.
  • Direct commits to main are heavily discouraged.

feature/* Branches

Each new piece of work, whether a new feature, bugfix, or chore, is done in a dedicated branch.

  • Created from the latest version of the main branch.
  • Branches should have descriptive names (e.g., feature/add-login, fix/header-bug).
  • Work is done in isolation, preventing unstable code from affecting main.
  • Once complete, the branch is merged back into main, often via a Pull Request.

The Role of Pull Requests

Pull Requests (or Merge Requests) are the cornerstone of the Feature Branch Workflow. Instead of merging directly, a developer opens a Pull Request to initiate a discussion about their changes before integrating them into the main branch.

Code Review

Allows team members to review and discuss the changes.

Automated Checks

Triggers automated builds and tests to run.

Controlled Merge

Prevents direct pushes and ensures quality control.
