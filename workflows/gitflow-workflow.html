Gitflow Workflow

A structured and robust Git branching model for managing larger projects. Gitflow is ideal for projects with a scheduled release cycle.

Visual Workflow Diagram

main Branch

The main branch (formerly master) stores the official release history. It should always reflect a production-ready state.

  • Contains stable, production code only.
  • Code is merged into main from release or hotfix branches.
  • Each commit on main is a new version and should be tagged (e.g., v1.0, v2.0.1).
  • Direct commits to this branch are strictly forbidden.

develop Branch

The develop branch serves as the integration branch for features. It contains the latest delivered development changes for the next release.

  • Represents the "next release" codebase. Can be unstable at times.
  • All feature branches are merged into this branch.
  • When ready for a release, a release branch is created from develop.
  • This is the main branch where developers collaborate.

Supporting Branches

feature/*

Used to develop new features. They branch off of develop and must merge back into develop. A feature branch should never interact directly with main.

release/*

Created from develop when it's ready for a release. This branch is for final testing, bug fixes, and preparing meta-data. Once ready, it's merged into both main (and tagged) and back into develop.

hotfix/*

Used for urgent fixes on a production release. They branch from main, and once the fix is complete, they are merged back into both main (with a new tag) and develop to ensure the fix isn't lost.
